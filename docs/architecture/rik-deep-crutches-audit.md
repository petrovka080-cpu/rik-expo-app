# RIK Platform: Deep Read-Only Audit of System Crutches

## 1. Executive Summary
В ходе глубокого read-only аудита кодовой базы RIK Platform выявлен высокий уровень технического долга, сосредоточенный преимущественно в слое взаимодействия с данными (UI ↔ DB) и генерации отчетов (PDF). 
Проект способен компилироваться и работать, однако его архитектура является хрупкой из-за массового сокрытия реальных контрактов БД (через `as any` и patch-скрипты), полного дублирования бизнес-логики между ролями и применения таймер-хаков для обхода гонок (race conditions) в интерфейсе. Основные риски лежат в невозможности безопасного масштабирования (добавления ролей, изменения таблиц) без провоцирования регрессий в непредсказуемых местах.

## 2. Карта костылей по категориям

### 2.1. Type Hacks & Schema Drift (P0)
* **Проявления:** Массовое использование `as any`, `as unknown as`, `@ts-ignore` в слое API (например, `api/_core.ts` и `api/director_reports.ts`). Наличие скриптов-мутаторов (`fix-ts-ignore.js`), которые перед билдом подавляют ошибки `tsc`.
* **Почему опасно:** Несоответствие `database.types.ts` реальному состоянию БД не отлавливается на этапе компиляции. Изменение типа колонки в PostgreSQL приведет к "тихому" падению UI в рантайме.
* **Масштаб:** Системно. Поражен весь слой сетевых запросов.

### 2.2. Runtime & UI-костыли (P1)
* **Проявления:** Использование `setTimeout` и `NEXT_TICK_HACK` для управления фокусом и состоянием модалок (например, `useAccountantKeyboard.ts`, модалки снабженца). 
* **Почему опасно:** Приводит к "залипанию" интерфейса на медленных устройствах (Android) или двойным вызовам (гонкам). 
* **Масштаб:** Локально, в сложных формах и модалках.

### 2.3. API / Data Coupling (Сцепка UI и БД) (P0)
* **Проявления:** Прямые вызовы `supabase.rpc(...)` и `supabase.from(...)` внутри `.tsx` файлов (экранов и модалок), например в `warehouse.tsx` и `DirectorDashboard.tsx`.
* **Почему опасно:** UI-компоненты напрямую зависят от названий таблиц и процедур БД. Отсутствие Service Layer делает невозможным переиспользование логики или написание юнит-тестов.
* **Масштаб:** Системно.

### 2.4. PDF-костыли (P1)
* **Проявления:** Ручная склейка гигантских HTML-строк в JS главном потоке (`pdfRunner.ts`). Дублирование блоков PDF-разметки (колонтитулы, подписи) между Складом и Директором.
* **Почему опасно:** Вызывает зависание интерфейса. При определенных объемах данных приводит к падению приложения (Hermes OOM / Address Size Fault).
* **Масштаб:** Изолированно в PDF-модулях, но критично для бизнес-потока.

### 2.5. Architectural & Role Duplication (P2)
* **Проявления:** Три физические копии экрана заявок: `ForemanSubcontractTab.tsx`, `BuyerSubcontractTab.tsx`, `AccountantSubcontractTab.tsx`.
* **Почему опасно:** Любая правка бизнес-правила (например, смена логики статуса "Ожидает") должна вноситься в 3 файла вручную. Системный риск рассинхронизации.

### 2.6. Debug Noise & Fallback Masking (P3)
* **Проявления:** Вызов `Alert.alert("Ошибка", e)` разбросан по UI-слою, выбрасывая пользователю `[object Object]`. `console.log` payload dumps на каждый рендер карточки подрядчика.
* **Почему опасно:** Засоряет логи, пугает пользователя невнятными сообщениями, вызывает лаги UI (при огромных дампах в консоль).

## 3. Точки концентрации (Hotspots)

1. **`app/(tabs)/warehouse.tsx` (>1500 LOC)**
   * **Проблема:** "Узел хаоса". Смешаны Data Fetch (Supabase), Local State, UI Renders, Modals и обработчики PDF.
   * **Статус:** Менять сейчас нельзя / Требует отдельной фазы (Декомпозиция).

2. **`src/lib/api/_core.ts` & `src/lib/supabaseClient.ts`**
   * **Проблема:** Точка входа в БД лишена типобезопасности. Касты `as any` пропускают битые данные в UI.
   * **Статус:** Можно позже. Требует генерации `database.types.ts` и ревью RPC.

3. **`src/lib/pdfRunner.ts`**
   * **Проблема:** Плотная связность генерации HTML, работы с файловой системой ОС (FS legacy) и Expo-Print.
   * **Статус:** Требует отдельного плана (Миграция на Expo 54 File API и удаление base64).

4. **Группа `*SubcontractTab.tsx` (Прораб, Снабженец, Бухгалтер)**
   * **Проблема:** 100% дублирование кода списков.
   * **Статус:** Требует отдельной фазы (Создание Shared Components).

## 4. Классификация по исправляемости

### A. Safe Cleanup (Чинить безопасно и немедленно)
* Удаление скриптов `fix-*.js` из `package.json` пайплайна.
* Обертывание всех `console.log` в `__DEV__` guard (Debug Gating).
* Нормализация текстов `Alert.alert("Ошибка")` -> безопасные Fallback UI-строки.
* Замена типа `"unknown"` на `"Без категории"` в `procurementTyping.ts`.

### B. Local Adapter Fix (Локально изолированные адаптеры)
* Скрытие вызовов `supabase.from()` в `warehouse.tsx` за адаптером `InventoryAdapter.ts` (без отрыва от текущего стейта).
* Захват сетевых ошибок в `api/_core.ts` через единый нормализатор, чтобы в UI не улетали объекты вместо строк.

### C. Structural Refactor Later (Опасно, нужен план)
* Декомпозиция `warehouse.tsx`.
* Объединение ролевых `...SubcontractTab.tsx` в единый разделяемый `SubcontractList.tsx`.
* Переписывание формирования PDF (`pdfRunner.ts`) на компонентный серверный рендеринг (`react-dom/server`).

### D. Keep Temporarily (Оставить как есть)
* Касты `as any` в `director.metrics.ts`, пока не будет обновлен `database.types.ts`.
* Логика Директора: Один экран "Контроль" с внутренними табами. Не выносить в отдельные экраны для сохранения изоляции роли.
* `setTimeout` на 500ms в `pdfRunner.ts` перед удалением кэша (до переписывания FS-модуля).

## 5. Самые опасные системные риски (Top 5)

1. **Мутаторы Type-Checking'а (`fix-ts-ignore.js`):** Скрывают расхождение между Frontend и Backend-схемами. Невозможно увернно рефакторить код, так как компилятор лжет об отсутствии ошибок.
2. **Прямое связывание UI и БД (Supabase In-Component):** Переименование одной колонки в БД требует изменения десятка React-компонентов.
3. **Строковая PDF Генерация:** Потребление оперативной памяти экспоненциально растет при увеличении таблиц. Риск стабильных падений приложения (OOM) на слабых устройствах.
4. **Duplication Ролевых Флоу:** Баг, найденный в флоу Снабженца, скорее всего присутствует у Прораба, но будет пропущен при локальном фиксе.
5. **No-Cancel Fetching (Stale-State):** В модалках каталога нет `AbortController`. Быстрый ввод текста или двойное нажатие вызывает гонку сетевых ответов и отображение неверных результатов.

## 6. Приоритетный Remediation Roadmap

### Phase 0: Pipeline & Visibility Sanity (Release Blockers)
* Удалить все JS-модификаторы исходного кода из пайплайна сборки.
* Убрать возврат `base64` из `expo-print` (фикс крашей iOS).
* Вычистить Production-шум и нормализовать алерты (Fail-safe alert wrappers).
* Настроить `database.types.ts` генерацию из Supabase.

### Phase 1: Local Decoupling & Typings (Tech Debt)
* Устранить `as any` в основных RPC-врапперах (`_core.ts`) с применением сгенерированных типов.
* Внедрить `AbortController` в поисковые поля и модалки.
* Вынести тяжелые DB-операции из UI компонентов в локальные хуки/адаптеры (но пока без полного переноса в сервисы).

### Phase 2: Services & Deduplication (Structural)
* Создание `src/services/` и полный перенос `supabase` вызовов из компонентов.
* Слияние `ForemanSubcontractTab`, `BuyerSubcontractTab` и `AccountantSubcontractTab` в единый UI-кит.
* Декомпозиция `warehouse.tsx` на атомарные контейнеры.

### Later: PDF & Caching
* Внедрение `React Query` или `Zustand` для кэширования профиля и справочников.
* Полный рерайт системы сборки PDF с конкатенации строк на React SSR-to-HTML.

## 7. Стабилизированные зоны и строгие запреты

### A. Что уже выглядит стабилизированным (Относительно безопасно)
1. **Базовая структура Навигации (Expo Router):** Роутинг между табами ролей и модульная изоляция директорий (например `screens/director/`) работают предсказуемо.
2. **Компоновка логики Директора:** Концепт "Директор = один монолитный экран 'Контроль' с под-состояниями" функционально изолирует его от других ролей и предотвращает утечку прав доступа.
3. **Supabase Auth Layer:** Инициализация сессии и управление токенами локализованы и не размазаны по приложению.

### B. Что нельзя трогать без отдельного плана (Не изменять!)
1. **Экран Директора:** Ни в коем случае не разделять Dashboard директора на множество экранов навигации. Сохранять архитектуру единого центра контроля.
2. **Экран Склада (`warehouse.tsx`):** Переписывать только по четкому плану (State -> Data -> Render). Случайное разделение сломает глубоко-связанные модалки.
3. **Native File System Layer (`expo-file-system/legacy`):** Замена логики путей и сохранения PDF приведет к потере доступа к ранее сгенерированным актам на устройствах пользователей. Требует осторожной миграции на SDK 54 API (`File`/`Paths`).
