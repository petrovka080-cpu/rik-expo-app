# Аудит UI/UX PDF Вьюера (Анализ "WhatsApp Full-Screen" эффекта)

## 1. Введение и Цель
Цель данного аудита — проанализировать текущую реализацию экрана `app/pdf-viewer.tsx` и выявить технические ограничения, которые мешают PDF-документу открываться по-настоящему в полноэкранном режиме с эффектом «парения» (как в WhatsApp или нативных iOS-приложениях, см. 3-й скриншот пользователя). 

Код **не изменялся**, предоставлен только анализ причин и механик.

---

## 2. Разбор текущей механики (Почему сейчас НЕ как в WhatsApp)

В текущей версии `pdf-viewer.tsx` визуальная область разделена жестко математически:

```tsx
// Строка 346+
const headerBarHeight = Platform.OS === "web" || width >= 768 ? 56 : 50;
const headerHeight = showChrome ? headerBarHeight + (Platform.OS === "web" ? 0 : insets.top) : 0;

// Строка 637
<View style={[styles.documentStage, { top: headerHeight }]}>{body}</View>
```

### Главный блокировщик эффекта "На весь экран": Жесткий `top`
Вы используете `position: "absolute"` для `documentStage` (сцены, где лежит WebView), но задаете ему отступ сверху: `top: headerHeight`. 
Что это значит:
1. PDF-документ **физически обрезан** сверху на высоту шапки (header) + челки телефона (insets.top). 
2. PDF не заходит "под" полупрозрачную шапку. Он всегда упирается в неё, как в бетонную стену.
3. В WhatsApp (и iOS QuickLook) документ **всегда занимает 100% экрана** (`top: 0`, `bottom: 0`), а полупрозрачная черная шапка (Header) просто **висит поверх него**.

### Проблема "Скачка" при прятании UI (Chrome Toggle)
В коде есть функция `toggleChrome`, которая скрывает и показывает `Header`.
Но так как `headerHeight` динамически меняется (от ~90px до 0px), скрытие интерфейса вызывает **физическое изменение размеров WebView**. 
Это приводит к:
* Перерисовке окна (layout thrashing).
* Скачку скролла внутри PDF (WebView дергается).
* Потере иллюзии плавности.

В идеальном приложении (WhatsApp) скрытие шапки просто делает ей `opacity: 0` (анимацией fade), а сам PDF-документ размер никак не меняет, так как он и так растянут на весь экран.

## 3. Анализ свойств самого `WebView` (Внутренние настройки)

В текущем конфиге `react-native-webview`:

```tsx
bounces={false}
contentInsetAdjustmentBehavior="never"
automaticallyAdjustContentInsets={false}
scalesPageToFit
```

### Что ограничивает "Нативное ощущение":
1. **`contentInsetAdjustmentBehavior="never"`** 
   В iOS эта настройка говоритWebView, чтобы он не пытался сам подложить паддинги под "челку" или шапку. В связке с растягиванием `top: 0` (для полноэкранности) текст документа может залезть прямо под часы телефона в самом начале. В нативных Viewers (WhatsApp) `WebView` растянут на весь экран, но ему передают `contentInset={{ top: 90 }}`, чтобы старт документа был сразу под шапкой, но при скролле вверх документ изящно уезжал *под* полупрозрачный Header.
2. **`bounces={false}`** 
   Отключает характерную для iOS "пружинку" при доскролливании до края. WhatsApp и нативный просмотрщик всегда "пружинят" PDF-ки, давая физическое ощущение документа.

## 4. Как лучшие приложения (WhatsApp / Telegram) делают этот UX

Чтобы добиться эффекта с 3-го скриншота, архитектура визуального слоя выстраивается по другому принципу:

1. **Базовый слой — 100% экрана:**
   `documentStage` занимает всё пространство от верхней кромки экрана (включая зону FaceID/часов) до самого низа: `top: 0, bottom: 0, left: 0, right: 0`.
2. **Отступы внутри PDF, а не снаружи:**
   Чтобы шапка сразу не перекрывала текст первой страницы, внутрь `WebView` прокидывается свойство `contentInset={{ top: headerHeight, bottom: indicatorHeight }}`. 
3. **Нависающий интерфейс (Chrome Overlays):**
   `Header` и `PageIndicator` лежат поверх `WebView` с `position: 'absolute'` и полупрозрачным `backgroundColor` (например, `rgba(17,17,17,0.8)` с эффектом blur).
4. **Бесшовный Toggle:**
   При клике по экрану, `Header` не удаляется из верстки и не меняет `top` документа. Он просто плавно исчезает через `Animated.View` (`opacity` от 1 к 0). Размер PDF-рамки остается статичным.

## 5. Выводы аудита (Что конкретно мешает текущей реализации)

1. Зависимость высоты PDF-сцены от состояния шапки (`top: headerHeight`).
2. Неиспользование внутренних `insets` самого `WebView` для компенсации перекрытия шапкой.
3. Отсутствие эффекта пружины (`bounces={false}`).
4. Механика Toggle, которая приводит к физическому ресайзу фрейма вместо наложения альфа-канала (Opacity / Fade).

**Резюме:** Сама логика загрузки и File-handling в файле сейчас написана отлично. Но визуальная верстка использует подход **"Flex/Stack"** (блоки стоят друг под другом), тогда как для премиального fullscreen-опыта нужен подход **"Z-Index Overlay"** (PDF занимает весь экран, а UI плавает поверх него).
