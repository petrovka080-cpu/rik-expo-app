# RIK Platform: Deep Read-Only Audit of System Crutches (v2)

## 1. Executive Summary (Текущее состояние)
В ходе обновленного read-only аудита кодовой базы RIK Platform (v2) зафиксировано существенное улучшение состояния проекта: внедрен строгий клиент Supabase, устранен ID/Schema drift, и проект успешно проходит проверку типов (`npx tsc --noEmit`). 

Тем не менее, в архитектуре остаются **активные костыли и структурные риски**, сосредоточенные в слое взаимодействия с данными (UI ↔ DB) и механизмах рендеринга PDF. Текущая архитектура стабильна на уровне компиляции, но ее дальнейшее масштабирование затруднено значительным дублированием бизнес-флоу между ролями и локальными хаками жизненного цикла (таймеры в модалках).

## 2. Что уже стабилизировано после Remediation

1. **Типобезопасность ядра БД:** Внедрен типизированный клиент Supabase, опирающийся на сгенерированные типы `database.types.ts`.
2. **Нулевой Type Drift:** Массовые несовпадения типов (Schema/ID drift) устранены. Проект компилируется "в зеленую" без скрывающих скриптов в пайплайне.
3. **Базовая структура Навигации:** Роутинг работает предсказуемо (через Expo Router).
4. **Изоляция Директора:** Логика Директора собрана в единый экран "Контроль" с внутренними табами, что предотвращает утечку прав и перекрестные зависимости с другими ролями.

## 3. Карта оставшихся костылей по категориям

### 3.1. API / Data Coupling (UI-сцепка с БД) (P1)
* **Проявления:** Прямые вызовы `supabase.from(...)` и `supabase.rpc(...)` остаются внутри `.tsx` файлов экранов (например, `warehouse.tsx` и графики в `DirectorDashboard.tsx`).
* **Почему опасно:** Компоненты UI продолжают "знать" о структуре БД. Отсутствие выделенного Service Layer затрудняет написание тестов и переиспользование запросов.
* **Масштаб:** Распространенный паттерн по всему приложению.

### 3.2. Runtime & UI-костыли (Timer Hacks) (P2)
* **Проявления:** Сохраняются обходные пути для управления фокусом и предотвращения гонок UI, такие как `setTimeout` и паттерны отложенного выполнения (`NEXT_TICK_HACK`) в модальных формах (например, `useAccountantKeyboard.ts`).
* **Почему опасно:** Приводит к нестабильной работе интерфейса на медленных устройствах (Android) или микрогонкам при быстрых нажатиях.
* **Масштаб:** Локализовано в сложных формах и модалках.

### 3.3. PDF-монолитность (P1)
* **Проявления:** Ручная конкатенация HTML-строк в JS главном потоке (`pdfRunner.ts`). Дублирование блоков разметки (header, footer, signatures) между различными ролевыми отчетами.
* **Почему опасно:** Вызывает резкие скачки потребления памяти и потенциальные падения (OOM) на мобильных устройствах при объемных отчетах. Усложняет добавление новых форматов документов.
* **Масштаб:** Изолированно в модуле генерации PDF.

### 3.4. Architectural Duplication (P2)
* **Проявления:** Наличие нескольких клонированных экранов субподрядов (`ForemanSubcontractTab.tsx`, `BuyerSubcontractTab.tsx`, `AccountantSubcontractTab.tsx`), реализующих схожий UI-флоу.
* **Почему опасно:** Риск рассинхронизации бизнес-логики. Обновление флоу требует ручного внесения изменений в несколько файлов.
* **Масштаб:** Ограничено ролевыми табами.

### 3.5. Debug Noise & Fallback Masking (P3)
* **Проявления:** Наличие агрессивных `console.log` (payload dumps) без dev-guard'ов, а также хардкод-заглушки вида `Alert.alert("Ошибка", e)` в catch-блоках.
* **Почему опасно:** Засорение логов в production, неинформативные уведомления для пользователей.
* **Масштаб:** Разбросано по обработчикам ошибок.

## 4. Точки концентрации (Hotspots)

1. **`app/(tabs)/warehouse.tsx` (>1500 LOC)**
   * **Тип проблемы:** Смешение слоев (Data Fetch, Local State, UI Renders, PDF Builders).
   * **Статус:** Менять сейчас нельзя / Требует отдельного четкого плана декомпозиции без смены логики.

2. **`src/lib/pdfRunner.ts`**
   * **Тип проблемы:** Монолитная строковая сборка интерфейса документа и работа с legacy-FS.
   * **Статус:** Менять сейчас нельзя / Требует поэтапного перехода на компонентную сборку (SSR).

3. **Группа `*SubcontractTab.tsx`**
   * **Тип проблемы:** Дублирование логики карточек и списков.
   * **Статус:** Можно позже. Требует постепенного выделения общих компонентов `src/components/shared/`.

---

## 5. Главные риски без изменения логики

1. **Прямая зависимость UI от DTO:** Любое изменение структуры таблиц потребует текстового поиска по всем компонентам.
2. **Дублирование запросов (Отсутствие кэша):** Переключение между экранами провоцирует повторные выгрузки справочников (каталог, подрядчики), создавая избыточную нагрузку на БД.
3. **Хрупкость PDF-генератора:** Рост объемов данных в актах может привести к отказам на стороне нативного моста из-за лимитов длины строк в Hermes.

## 6. Классификация по исправляемости (Action Plan)

### A. Safe Cleanup (Безопасно для немедленного исправления)
* Устранение Debug-шума: Обертывание `console.log` в `__DEV__`.
* Нормализация текстов `Alert.alert("Ошибка")` -> внедрение безопасных Fallback UI-строк и тостов.
* Очистка неактуальных комментариев и мертвых импортов, оставшихся после `tsc` cleanup.

### B. Local Adapter Fix (Обертки без смены контракта)
* Создание хуков-адаптеров для вызовов Supabase в тяжелых компонентах (первый шаг к Service Layer).
* Изоляция повторяющихся маппингов данных в `useMemo` или хуки внутри ролевых экранов.

### C. Structural Refactor Later (Требует архитектурной фазы)
* Выделение чистого Service Layer (`src/services/`) и вынос туда всех RPC/Direct вызовов.
* Постепенное объединение ролевых `...SubcontractTab.tsx` в единый разделяемый UI-шаблон.
* Миграция PDF-модуля на компонентный рендеринг (React SSR) и файловое API Expo 54.

### D. Keep Temporarily (Не трогать / Оставить как есть)
* **Архитектура Директора:** Директор ДОЛЖЕН оставаться единым экраном "Контроль" с внутренними вкладками. Строго запрещено дробить роль директора на разные навигационные экраны.
* Локальные `useEffect` с ручным стейтом загрузки (до внедрения `React Query` или аналогичного кэша).
* `setTimeout` хаки в модальных компонентах дна BottomSheet (пока не будет выделено время на глубокий рефакторинг UI-потока клавиатуры).
