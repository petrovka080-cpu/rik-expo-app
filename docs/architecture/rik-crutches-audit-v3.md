# Глубокий аудит костылей RIK Platform (v3.1 Final)

## 1. Executive Summary

В результате строгого read-only аудита установлено, что RIK Platform находится на стадии уверенной стабилизации базовых контрактов. Проект успешно преодолел этап хаотичной типизации — внедрен `database.types.ts`, устранен `schema drift`, а пайплайн сборки очищен от мутаторов. Также начата активная работа по очистке "моджибаке" (исправлены кодировки в `files.ts`). 

Однако, проект по-прежнему несет в себе критический объем архитектурного долга, препятствующего горизонтальному масштабированию. Основные риски лежат в жесткой связности UI с базой данных (вызовы `supabase` напрямую из JSX), хрупкости монолитного PDF-движка, а также тотальном копипасте ролевых процессов (например, вкладки субподрядов). Наличие таймер-хаков (обходы race conditions) в модальных окнах снижает стабильность UI на Android.

**Общий уровень техдолга:** Средне-высокий (сместился от критического благодаря типизации).
**Главные риски:** Затрудненное изменение схемы БД из-за размазанных по UI SQL/RPC запросов, падения Hermes (OOM) при сборке гигантских PDF.

## 3. Карта костылей

### 3.1. UI ↔ DB Coupling
* **Уровень риска:** P0
* **Где встречается:** `app/(tabs)/warehouse.tsx`, экраны Директора, `buyer.actions.ts`.
* **Масштаб:** Системный по всему приложению.
* **Симптомы:** React компоненты делают напрямую `await supabase.from('inventory').select()`. Вьюха жестко привязана к физической таблице.

### 3.2. PDF Issues (Монолитность стрингов)
* **Уровень риска:** P1
* **Где встречается:** `src/lib/pdfRunner.ts`, `src/lib/api/pdf.ts`.
* **Масштаб:** Изолирован, но критичен для бизнес-флоу.
* **Симптомы:** Формирование UI актов через ручную конкатенацию HTML в строковых переменных (`html += ...`). Дублирование кусков разметки и inline-стилей Tailwind.

### 3.3. Архитектурное дублирование (Ролевой копипаст)
* **Уровень риска:** P1
* **Где встречается:** `ForemanSubcontractTab.tsx`, `BuyerSubcontractTab.tsx`, `AccountantSubcontractTab.tsx`.
* **Масштаб:** Широкий (все ролевые экраны с общим контекстом).
* **Симптомы:** Три копии одного и того же UI-списка заявок и статусов, делающие одинаковые запросы. 

### 3.4. Runtime-костыли (Race Condition Hacks)
* **Уровень риска:** P2
* **Где встречается:** `useAccountantKeyboard.ts`, модалки выбора снабженца.
* **Масштаб:** Локально в сложных попапах/шторках (BottomSheets).
* **Симптомы:** `setTimeout(() => ref.current.focus(), 100)` для предотвращения зависания интерфейса между перерисовкой слоев ОС.

### 3.5. Type Hacks (Специфичные касты)
* **Уровень риска:** P2
* **Где встречается:** Внутри парсеров RPC-ответов (например `api/_core.ts`).
* **Масштаб:** Локализован.
* **Симптомы:** Единичные `as any` или `as unknown as` для принудительного приведения вложенных JSON-компонентов.

### 3.6. Debug Noise & Fallback masking
* **Уровень риска:** P3
* **Где встречается:** Утилиты, логика карточек (`contractor.viewModels.ts`).
* **Масштаб:** Точечный.
* **Симптомы:** Незакрытые `console.log(payload)` на этапе продакшена, вызывающие падение FPS в длинных списках.

## 4. Hotspots (Зоны риска)

1. **`app/(tabs)/warehouse.tsx` (Склад)**
   * **Размер:** >1500 строк.
   * **Тип проблемы:** "Узел хаоса". Сборник API-вызовов, модалок, локального стейта и генератора подписей для PDF. 
   * **Риск:** Запредельный при внесении новых фич.

2. **`src/lib/pdfRunner.ts` (Генератор актов)**
   * **Размер:** Монолитный движок.
   * **Тип проблемы:** Паровоз строк HTML + файловые legacy-операции ОС.
   * **Риск:** Риск нехватки памяти (OOM Size Limit) на мобильных устройствах.

3. **Семья `*SubcontractTab.tsx`**
   * **Размер:** 3-4 файла по 500-1000 строк.
   * **Тип проблемы:** Ctrl+C / Ctrl+V архитектура.
   * **Риск:** Десинхронизация логики между ролями (забыли добавить статус у Бухгалтера, но добавили у Прораба).

## 5. Топ 5 системных рисков (Блокировщики масштабирования)

1. **Supabase Vendor Lock внутри JSX.** (Невозможность гибкого кэширования профиля/каталога без Service Layer).
2. **Память Hermes JS Threads.** (Строковая сборка тяжелых PDF приводит к Address Size Fault).
3. **Ролевой монолитизм.** (Поддержка карточек Подрядов у 3-х ролей требует в 3 раза больше времени разработчика).
4. **Таймеры.** (Использование `setTimeout` вместо событий жизненного цикла ломает предсказуемость UI на слабых аппаратах).
5. **N+1 Fetching.** (Компоненты монтируются и делают локальные `useEffect` вызовы каталога вместо получения его из единого store).

## 6. Классификация по исправляемости

### Safe cleanup (Можно убрать без изменения логики)
* **Production Logs:** Обернуть `console.*` в блоки `if (__DEV__)`.
* **Mojibake Fixes:** Продолжить вычищать Alert-ы и fallback тексты, как это было сделано в `files.ts`.

### Local adapter fix (Можно изолировать адаптерами)
* **Type Normalization:** Последние `as any` в `api/_core.ts` можно закрыть строгими Type Guards.
* **Supabase Queries:** Вынести логику `supabase.from()` в соседние хуки (например, `useWarehouseItems()`), чтобы разгрузить сам `.tsx` визуальный рендер.

### Structural refactor later (Нужен отдельный план)
* **Service Layer:** Папка `src/services/` для всех сетевых сущностей.
* **Shared Components:** Объединение флоу Подрядов в единый `SubcontractList`.
* **PDF React SSR:** Замена строковой конкатенации на генерацию HTML через React Components (`react-dom/server`).
* **React Query / Zustand:** Кэширование запросов профиля и каталога товаров на уровне всего App.

### Keep temporarily (Лучше пока не трогать)
* **Роль Директора:** Монолитный экран "Контроль" (не дробить на новые экраны).
* **Модальные таймеры:** Оставить `setTimeout` хаки до масштабной рефакторизации UI Keyboard flows.
* **FS Legacy:** Ждать полной миграции на File Paths из Expo 54.
