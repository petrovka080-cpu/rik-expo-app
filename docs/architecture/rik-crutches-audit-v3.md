# Глубокий аудит костылей RIK Platform (v3)

## 1. Executive Summary

В результате строгого read-only аудита установлено, что RIK Platform находится в промежуточной стадии стабилизации. Базовые проблемы типо- и схема-рассинхронизации (schema drift) в значительной мере решены: внедрен типизированный клиент Supabase, а кодовая база успешно проходит проверку типов (`tsc`). 

Тем не менее, проект содержит критический объем архитектурного долга, который препятствует его безопасному масштабированию. Основные риски сконцентрированы в глубокой связности UI с базой данных (прямые SQL/RPC вызовы из React-компонентов), монолитности и хрупкости алгоритмов генерации PDF, а также в массовом дублировании бизнес-логики между экранами различных ролей. Наличие "time-хаков" в рантайме маскирует гонки (race conditions) при отрисовке UI. 

**Общий уровень техдолга:** Высокий (структурный).
**Главные риски:** Невозможность расширения схемы БД без каскадного рефакторинга UI, падения приложения (OOM) при генерации объемных PDF.

---

## 2. Что уже стабилизировано

Перед тем как разбирать костыли, важно отметить архитектурные успехи. Следующие зоны приведены в безопасное состояние:

1. **Typed Supabase & Schema Drift:** Ядро БД работает через `database.types.ts`. Массовые расхождения (`schema drift`) устранены. 
2. **Нулевой Type Drift в Pipeline:** Устранены мутирующие pre-build скрипты. Проект компилируется (`tsc --noEmit` is green) по-честному, без внешнего подавления ошибок.
3. **Навигация (Expo Router):** Базовая навигационная структура табов и роутингов работает предсказуемо и стабильно.
4. **Изоляция Директора:** Директор функционирует как единый экран "Контроль" с внутренними табами, что архитектурно правильно изолирует эту бизнес-роль от случайного переиспользования UI рядовых сотрудников.

## 3. Карта костылей

### 3.1. UI ↔ DB Coupling (Сцепка интерфейса и данных)
* **Уровень риска:** P0 (Критический)
* **Где встречается:** `app/(tabs)/warehouse.tsx`, `DirectorDashboard.tsx`, хуки `buyer.actions.ts`.
* **Масштаб:** Системный.
* **Описание:** React-компоненты напрямую используют `supabase.from(...)` и `supabase.rpc(...)`. Если изменится название колонки или таблицы, перестанет работать рендер UI, и это потребует глубокого поиска по исходникам.

### 3.2. PDF Issues (Монолитность и String-сборка)
* **Уровень риска:** P1 (Высокий)
* **Где встречается:** `src/lib/pdfRunner.ts` и `api/pdf.ts`, `warehouse.tsx` (Signatures).
* **Масштаб:** Изолирован в слое PDF.
* **Описание:** Огромные HTML-строки формируются ручной конкатенацией (`+`). Повторяющиеся блоки (headers/footers/signatures) дублируются в разных отчетах. Обработка гигантских строк в основном JS-потоке тормозит UI и провоцирует OOM (Out Of Memory) краши.

### 3.3. Архитектурное дублирование (Role Duplication)
* **Уровень риска:** P1 (Высокий)
* **Где встречается:** `ForemanSubcontractTab.tsx`, `BuyerSubcontractTab.tsx`, `AccountantSubcontractTab.tsx`.
* **Масштаб:** Во всех модулях, зависящих от бизнес-сущности подрядов.
* **Описание:** 100% дублирование кода карточек, статусов и флоу списков. Риск заключается в том, что исправление бага или добавление нового статуса у Прораба не будет автоматически перенесено Снабженцу.

### 3.4. Runtime-костыли (Timer-хаки)
* **Уровень риска:** P2 (Средний)
* **Где встречается:** `useAccountantKeyboard.ts`, модалки снабженца, `pdfRunner.ts`.
* **Масштаб:** Локально в сложных модалках.
* **Описание:** Использование `setTimeout` / `NEXT_TICK_HACK` для принудительной передачи фокуса или скрытия Race Conditions между закрытием BottomSheet и открытием клавиатуры.

### 3.5. Type Hacks (Остаточное подавление типов)
* **Уровень риска:** P2 (Средний)
* **Где встречается:** `api/_core.ts`, парсеры в `ForemanSubcontractTab.tsx`.
* **Масштаб:** Локализованные cast-адаптеры.
* **Описание:** `as any` и `as unknown as` всё ещё применяются для обхода строгой схемы при парсинге вложенных JSON-данных из RPC, возвращая "слепые" объекты в UI.

### 3.6. Debug Шум (Production Noise)
* **Уровень риска:** P3 (Низкий)
* **Где встречается:** Карточки контрагентов, глобальные обработчики ошибок.
* **Масштаб:** Точечный.
* **Описание:** Спам `console.log` на каждый рендер сложного списка (убивает производительность JS thread). Использование `Alert.alert("Ошибка", e)` выводит пользователю страшный текст `[object Object]`.

## 4. Hotspots (Точки концентрации)

1. **`app/(tabs)/warehouse.tsx`**
   * **Размер:** >1500 строк.
   * **Тип проблемы:** Смешение UI + API (Supabase) + Local State + PDF Builders.
   * **Риск:** Высокий. Файл слишком хрупок для внедрения нового функционала (любая правка может нарушить работу смежных модалок).

2. **`src/lib/pdfRunner.ts`**
   * **Размер:** Монолит генерации.
   * **Тип проблемы:** Строковые манипуляции, смешанные с файловыми ОС-операциями.
   * **Риск:** Высокий. Основной кандидат на сбои (Memory size limits).

3. **`*SubcontractTabs` (Группа файлов)**
   * **Размер:** 500-1000 строк каждый.
   * **Тип проблемы:** Физическое клонирование бизнес-логики.
   * **Риск:** Высокий риск регрессионных багов из-за рассинхронизации.

---

## 5. Топ системных рисков (5 главных проблем)

1. **Намертво вшитый Supabase в UI-слой.** Блокирует безопасный рефакторинг БД.
2. **Отсутствие кэширования профиля и справочников.** Мобильное приложение делает N+1 запросов при каждом переключении экранов, перегружая БД.
3. **Хрупкость генератора PDF.** Строковый подход упирается в лимиты памяти мобильных устройств.
4. **Role Flow Duplication.** Удорожает поддержку платформы в 3-4 раза.
5. **Таймеры маскировки (Runtime Hacks).** Скрывают реальные сбои жизненного цикла React, что приводит к невоспроизводимым багам UI.

## 6. Классификация по исправляемости

### Safe cleanup (Можно убрать без изменения логики)
* **Очистка Console:** Обертывание `console.*` в `if (__DEV__)`.
* **Устранение Mojibake/Alerts:** Замена дефолтных `Alert.alert("Ошибка", e)` на безопасные fallback-строки.
* **Удаление мертвого кода:** Чистка файлов от старых закомментированных импортов и скриптов-мутаторов.

### Local adapter fix (Можно изолировать адаптерами)
* **Type Normalization:** Замена `as any` в `api/_core.ts` на использование Zod / строгих типизированных адаптеров.
* **Supabase Wrappers:** Вынос вызовов `supabase.rpc()` изнутри JSX в кастомные хуки (например, `useDirectorMetrics()`) в тех же файлах для первичной изоляции.

### Structural refactor later (Нужен отдельный план)
* **Service Layer:** Создание папки `src/services/` и полный перенос работы с БД туда.
* **Component Reuse:** Переписывание SubcontractTabs на базе единого `SharedSubcontractList`.
* **PDF Engine:** Перевод отчетов на `react-dom/server` (SSR to HTML).
* **Global Cache:** Интеграция `React Query` (или `SWR`) для кэширования каталога.

### Keep temporarily (Лучше пока не трогать)
* **Директор:** Экран "Контроль" должен остаться монолитным. Строго не трогать и не дробить!
* **Keyboard/Modal `setTimeout`:** Оставить до момента системного внедрения BottomSheet SDK, иначе начнут залипать клавиатуры.
* Устаревший API файловой системы `expo-file-system/legacy` в PDF (оставить до полного перехода на Экспо 54 File API).

## 7. Roadmap (План исправлений без смены логики)

* **Phase 0 — Безопасный cleanup**
  * Исправление всех моджибаке, "кракозябр", пользовательских ошибок (Alert overrides). Защита production логов (`__DEV__`).
* **Phase 1 — Адаптеры (Types & Hooks)**
  * Локальный вынос вызовов Supabase из компонентов в хуки. Устранение `as any` через типизированные wrappers внутри API-файлов.
* **Phase 2 — Сервисный слой и Кэш**
  * Внедрение `src/services/`. Подключение `React Query` для предотвращения дублирования запросов справочников. Настройка единого слоя Data Fetching.
* **Later — Архитектурные изменения**
  * Переход на `react-dom/server` для PDF. 
  * Глубокая декомпозиция `warehouse.tsx`. 
  * Объединение интерфейсов подрядов (`SubcontractTabs`) в универсальные Shared компоненты.

---

## 8. Что нельзя трогать без плана (Особые зоны)

1. **Director Control Screen:** Архитектура монолитного экрана с внутренними табами идеальна для данной роли. **Запрещено** разбивать её на отдельные навигационные сущности ролей.
2. **`warehouse.tsx`:** Декомпозиция этого "гиганта" без детального ТЗ по стейт-менеджменту приведет к поломке модальных окон инвентаризации.
3. **PDF FS Layer (`pdfRunner.ts`):** Таймеры и способы очистки файлов в этом классе жестко сцеплены с багами `expo-print`. Любое "улучшение" без нативных стендов грозит черными экранами при попытке шеринга.
