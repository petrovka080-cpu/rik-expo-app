# Экстремально глубокий Read-Only аудит RIK Platform (Evidence-Based V5)

## Часть A. Executive Summary
**Состояние проекта:** RIK Platform прошел фазу "первичной очистки" (schema drift устранен, `tsc` компилируется без подавления ошибок). Внедрение `database.types.ts` и локальных адаптеров (на примере `buyer.counterparty.data.ts`) создало островок предсказуемости в хаосе.
**Главный архитектурный диагноз:** Архитектура всё ещё "спаяна" по вертикали. Бизнес-логика, структура БД и рендеринг находятся в одних и тех же или соседних файлах. 
**Эффект последних Batch-уборок:** Изменения *реально* улучшили безопасность типов и обработку краевых случаев (типа "моджибаке" в алертах файлов или fallback-данных у снабженца). Однако, они действуют как "локальные пластыри". Техдолг часто не устраняется, а инкапсулируется в сложные `compat`-слои и `use...Adapter` хуки, которые разрастаются.
**Главная угроза масштабируемости:** Отсутствие Service Layer (промежуточного кэширующего и сервисного API) и N+1 загрузки гигантских массивов в рантайме React Native.

## Часть B. Updated Technical Debt Map (Текущая реальность)

### P0 (Блокировщики архитектурного развития)
1. **Screen-level Supabase/RPC Coupling**
   * **Где:** `app/(tabs)/warehouse.tsx`, логика директора.
   * **Масштаб:** Десятки вызовов компонентов напрямую в БД.
   * **Влияние на масштабирование:** Невозможно безопасно изменить схему таблиц или перевести часть логики на отдельный бекенд. Любая сменя логики требует точечного поиска UI-файлов.
   * **Что осталось:** Проблема не решена, вызовы лежат в визуальных слоях.
2. **Duplicated Role Logic (Ctrl+C Архитектура)**
   * **Где:** `ForemanSubcontractTab.tsx` и его клоны у Снабженца и Бухгалтера.
   * **Влияние на масштабирование:** Внесение нового статуса требует написания одного и того же кода в 3-х местах. Утроенный риск регрессий.

### P1 (Риски отказоустойчивости)
3. **Heavy Fan-out & Startup Orchestration Debt**
   * **Где:** Хуки загрузки каталога и профиля (`buyer.counterparty.data.ts`).
   * **Масштаб:** При открытии экрана грузятся тысячи записей (3000 contractors, 2000 subcontracts) параллельно через `Promise.allSettled`.
   * **Влияние на масштабирование:** При росте количества поставщиков телефон зависнет или упадёт от OOM (Out of Memory) в основном JS-потоке.
   * **Что улучшено:** Загрузка стала типобезопасной и получила fallback-механизмы.
4. **PDF Fragility (Монолитность отчетов)**
   * **Где:** `pdfRunner.ts`.
   * **Влияние:** Строковая сборка актов. При больших объемах данных `expo-print` будет аварийно завершаться.

### P2 (Скрытые баги и гонки)
5. **`setTimeout` / Race Hacks**
   * **Где:** Более 40 вхождений в модалках и `Keyboard` обработчиках (`ActivePaymentForm.tsx`, `warehouse.utils.ts`).
   * **Влияние:** Скрывает гонки UI (Race Conditions). Вызывает "фантомные" залипания для пользователей на старых устройствах Android.
6. **Fallback-heavy Compat Code / Type Hacks**
   * **Где:** Парсинг JSON-полей внутри компонентов (`(payload.data as any).request_no`).
   * **Влияние:** Компилятор считает код "зеленым", но в рантайме это упадет, если поле изменит структуру.

### P3 (Шум и DX)
7. **Dev-noise в Hot Path**
   * **Где:** Разбросанные по всему коду `console.log(payload)`, не закрытые `__DEV__` проверками.
   * **Влияние:** Просадки FPS при скроллинге списков.
   * **Что улучшено:** Выполнен частичный cleanup моджибаке и ошибок ФС (например, `files.ts`).

## Часть C. Assessment of recent changes (Оценка уже сделанных чисток)

1. **Buyer cleanup / startup diagnostic / local adapterization (`buyer.counterparty.data.ts`)**
   * **Что реально улучшено:** Убран слепой маппинг `as any`. Внедрена строгая архитектура загрузки (Load Plans). Появилась локальная диагностика загрузки поставщиков.
   * **Что осталось:** Тяжелые `limit(3000)` селекты. При загрузке тянется избыточный вес данных в главный поток.
   * **Не появился ли новый compat debt:** Появился. Логика `fetchBuyerContractorsFallback` маскирует проблемы с основной схемой `select("id,company_name...")`.
   * **Итоговая оценка:** *Partial cleanup / Debt-shifting fix.* Логика стала безопасной, но переросла в тяжелый data-fetching узел на стороне клиента (должно быть на бекенде или в сервисе).

2. **Director safe-cleanup batch / Workflow restrictions**
   * **Что реально улучшено:** Логика директора зажата в рамках одного таба `Контроль`. Отсутствие `router.push` на чужие экраны сдерживает "утечку ролей".
   * **Что осталось:** Финансовые функции и графики все еще зависят от монолитного файла данных.
   * **Итоговая оценка:** *Good safe cleanup.* Изоляция роли спасает проект от хаоса навигаций.

3. **Files mojibake / Alert cleanups (`files.ts`)**
   * **Что реально улучшено:** "Кракозябры" UI устранены. Имплементированы безопасные fallback-тексты на кириллице. Учтена специфика `window.prompt` при блокировке поп-апов в браузере.
   * **Итоговая оценка:** *Good safe cleanup.* Улучшило UX без изменения контрактов БД.

## Часть D. Hotspot ranking (Самые опасные зоны)

1. **`app/(tabs)/warehouse.tsx` (Складской хаб)**
   * **Почему опасен:** Монолитный файл (1500+ строк), в котором пересекаются UI, вызовы RPC для списания/приемки и конфигурация модалок. 
   * **Почему блокирует:** Невозможно добавить новый бизнес-процесс склада (например, инвентаризацию с штрихкодами), не рискуя сломать текущую приемку.
   * **Почему нельзя оставлять:** Это центральная вена движения материальных ценностей компании.

2. **Группа `*SubcontractTab.tsx` (Флоу Субподрядов)**
   * **Почему опасен:** "Copy-Paste" архитектура. Логика статусов и UI списков размножена по 3 ролям.
   * **Почему блокирует:** Внедрение нового стейта заявки (например, "На согласовании у инженера") потребует ручного изменения 3-х не связанных напрямую файлов.

3. **Dataloaders / N+1 Hooks (Хуки вроде `useBuyerSuppliers`)**
   * **Почему опасен:** Fetching 2000-3000 строк из БД напрямую на мобильном клиенте.
   * **Почему нельзя оставлять:** Телефонная память (Hermes RAM) не резиновая. Бизнес растет, количество строк растет — приложение начнет "схлопываться" при старте.

## Часть F. Workflow integrity risks (Риски утечки логики ролей)

* **Устойчивость Директора:** Высокая. Изоляция в "Контроль" экране надежна, пока никто не добавит туда прямые переходы (`router.push`) в другие секции.
* **Скрытые риски Subcontract Flow:** Логика согласования "размазана" между Прорабом и Бухгалтером. Поскольку каждый экран делает *свои* прямые апдейты в БД (через `supabase.from("requests")`), нет единого State Machine (автомата состояний). Прораб теоретически может внедрить апдейт, недопустимый с точки зрения Бухгалтера.

---

## Часть G. Runtime / stability risks (Производительность)

* **Serial chains & Heavy loops:** Анализ показал, что компоненты маппят тысячи объектов при загрузке (fan-out давление на JS-поток).
* **`setTimeout` Gaps:** Множественные хаки синхронизации модалок. Риск "State Update on Unmounted Component", если юзер закроет экран быстрее, чем сработает таймер (500мс).

## Часть H. Prioritized No-Code Roadmap (Что делать дальше)

**НЕ "переписывать все", а поступательно развязывать узлы.**

* **Batch 1: Isolation & Type Propagation (Заморозка костылей)**
  * *Цель:* Запретить появление новых вызовов `supabase.from` в UI.
  * *Действие:* Тотальный вынос оставшихся прямых вызовов БД из JSX в локальные изолированные хуки (как сделали с `useBuyerSuppliers.ts`, только без увеличения нагрузки).
  * *Побочное действие:* Дочистить остатки `__DEV__` консоль-логов.
* **Batch 2: Global State & Memory Relief (Кэширование)**
  * *Цель:* Убить N+1 загрузку тысяч строк при смене роутов.
  * *Действие:* Внедрение React Query (или Zustand) для `Справочников`, `Профилей`, `Контрагентов`. Клиент должен грузить их 1 раз, а не 5 раз параллельно.
* **Batch 3: Shared UI De-duplication**
  * *Цель:* Избавиться от размноженных ролевых вьюшек.
  * *Действие:* Создание `src/components/shared/SubcontractList.tsx` и замена карточек у Buyer, Foreman, Accountant на этот компонент.
