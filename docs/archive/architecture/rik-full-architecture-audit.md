# RIK Platform: Полный Архитектурный Аудит и Карта Технического Долга

## 1. Executive Summary

**Текущее состояние архитектуры:** 🔴 КРИТИЧЕСКОЕ (Высокий риск масштабируемости и стабильности).
Проект находится на стадии накопленного технического долга. Текущая кодовая база представляет собой набор переплетенных слоев (UI, State, FileSystem, Network), что делает невозможным безопасное масштабирование или добавление новых ролей (Accountant) без регрессий в других частях системы.
**Общий уровень риска:** Высокий. Система содержит нестабильные Runtime-хаки, "угадывание" типов данных из БД и прямое мутирование кода при сборке, что полностью нивелирует преимущества TypeScript и статической типизации.

## 2. Таблица проблем (Критически важные уязвимости)

| Severity | Type | File/Domain | Issue | Risk |
|---|---|---|---|---|
| 🔥 CRITICAL | FileSystem | `pdf.ts`, `pdfRunner.ts` | Смешивание `legacy` и современных API Expo на фоне iOS 18 Security policies | Нативные краши (SIGABRT/OOM) в WebView/Hermes |
| 🔥 CRITICAL | Build Hack | `fix-*.js` scripts | Мутация TypeScript кода через Regex перед сборкой (auto-injection `@ts-ignore`) | Скрытые баги в Production, потеря статического анализа |
| 🚨 HIGH | API/Supabase | `_core.ts`, `*.api.ts` | Прямые RPC вызовы в связке с кастованием `as any` и `as unknown as` | Поломка экранов при малейшем изменении схемы БД |
| 🚨 HIGH | Runtime/State | `BuyerDropodown`, Modals | Prop drilling и гонки состояний (Race conditions) в `useEffect` | Рассинхрон UI (дрожание клавиатуры, потеря стейта дропдаунов) |
| 🚨 HIGH | Memory/Storage| `fileSystemPaths.ts` | Бесконтрольный рост `cacheDirectory` без garbage collector'a генерации PDF | Засорение памяти устройств, OOM-краши |
| ⚠️ MEDIUM | Architecture | `app/(tabs)/*` | "Fat Components": сетевые запросы, стейт и верстка в одном файле `>1000` строк | Невозможность переиспользования UI, "эффект домино" при фиксах |
| ⚠️ MEDIUM | PDF Engine | `pdfFileContract.ts` | Написание HTML-отчетов через конкатенацию гигантских строк в JS-потоке | Замораживание UI-потока, краш движка Hermes (String Limit) |

## 3. Top-10 самых опасных мест системы

1. **Генератор PDF-файлов (`src/lib/api/pdf.ts`)** - Неконтролируемая аллокация памяти. Возврат гигантских Base64 строк в JavaScript-среду и попытки их хешировать убивают движок Hermes (краши, которые вы сейчас лечите).
2. **Скрипты кодогенерации/патчинга (`fix-ts-ignore.js`, `fix-stars.js`)** - Это бомба замедленного действия. Скрипт просто глушит ошибки TypeScript, скрывая реальные несовпадения контрактов с БД.
3. **Модуль `pdfDocumentSessions.ts`** - Попытка управлять жизненным циклом физических файлов на устройстве с помощью React-парадигм.
4. **Механизм RPC-запросов к Supabase** - Угадывание camelCase/snake_case ключей из БД прямо внутри UI-компонентов.
5. **Мега-компоненты экранов (`warehouse.tsx`, `Contractor Screen`)** - Логика складирования, получения данных, отрисовки карточек и генерация PDF слеплены в `useEffect` макароны.
6. **Отсутствие Request Manager'a** - Компоненты сами "решают" когда делать API fetch, при размонтировании нет отмены запросов (AbortController), что приводит к state memory leaks.
7. **Legacy FileSystem Bridge** - Использование `expo-file-system/legacy` работает на старом мосту Objective-C, который не успевает за быстрой сменой контекста в React Native на новых ОС.
8. **Компоненты модальных окон (Buyer Editor)** - Перерисовка всего дерева компонента при каждом вводе символа в текстовое поле.
9. **Обработка подписок/Subcontracts** - Дублирование одного и того же флоу (Foreman, Accountant, Warehouse) вместо использования единого контроллера бизнес-логики.
10. **Обработка временных файлов WebView (`pdf-viewer.tsx`)** - Попытка прокинуть file:// URI в веб-вью до того, как оперативная память сбросила файл на жесткий диск устройства (File Flush latency).

## 4. Архитектурные костыли (Детализация)

### Слой: FileSystem
* **Уязвимость:** Использование `copyAsync` и `writeAsStringAsync` без проверки того, жив ли еще компонент, который их вызвал.
* **Риск:** Если пользователь нажал "открыть PDF" и сразу закрыл шторку/перешел на другой экран, promise попытается обновить размонтированный стейт или запустить WebView с пустой ссылкой.
* **Костыль:** Переход между `/legacy` и обычным API в попытках обойти желтые варнинги, сохраняя при этом несовместимость с нативным скомпилированным приложением.

### Слой: PDF Engine
* **Уязвимость:** `html` string builder. Бизнес-логика (например, подсчет тоталов или фильтрация позиций) выполняется *прямо внутри* функции генерации HTML текста. 
* **Риск:** Невозможно протестировать бизнес-правила отдельно от генерации PDF.
* **Костыль:** `hashString32(html)`. Попытка сделать "умное" кэширование через хеширование десятков мегабайт текста в синхронном JS треде.

### Слой: Supabase / API
* **Уязвимость:** Полное отсутствие DTO (Data Transfer Objects) или Zod/Yup валидаторов.
* **Риск:** Если Supabase вернет `null` в поле `price`, UI-слой просто упадет с `Cannot read property of undefined` во время рендера.
* **Костыль:** Использование TypeScript подавителей и `as any`, чтобы компилятор не мешал собирать проект.

### Слой: UI & State
* **Уязвимость:** Прокидывание сеттеров состояний (`setIsOpen`, `setItems`) на 4-5 уровней вглубь дерева (Prop Drilling).
* **Риск:** Любой дочерний компонент может нарушить консистентность главного экрана. 
* **Костыль:** Гонки эффектов (`useEffect` внутри `useEffect`). Например, дропдаун ломается, потому что один эффект ставит значение по умолчанию, а другой (вызванный на милисекунду позже сеткой) — его затирает.

## 5. Масштабируемость

Текущая архитектура **не пригодна** для безопасного добавления новых пользовательских ролей без копипаста. 

* **Роли (Roles):** Архитектура построена по принципу "Скопировать Warehouse Tab и переименовать в Foreman Tab". Это приведет к тому, что исправление одного бага (например, расчета сумм) придется дублировать в 5-ти разных файлах руками.
* **Отчеты (Reports):** Изоляция генераторов отчетов. Директорский и Складской отчет написаны по-разному, хотя используют один и тот же PDF движок.
* **Хранилище (Storage):** При добавлении большого количества фотографий или чеков `cacheDirectory` быстро переполнится программным мусором, устройство пользователя начнет зависать.

## 6. План remediation (Дорожная карта восстановления)

* **Phase 0 – Crash containment (Контейнеризация крашей):**
  - Выпустить новый нативный билд (EAS Build IPA/AAB) с Expo SDK 54.
  - Убрать Base64 генерацию PDF, перейти на файловое API Expo 54 (`File.copy`).
  - Убрать тяжелое хеширование (`hashString32`) из UI потока.
* **Phase 1 – Infrastructure stabilization (Стабилизация):**
  - Удалить ВСЕ скрипты `fix-*.js`. Зафиксировать типы вручную через генератор типов Supabase API.
  - Внедрить Zod/Joi для валидации данных, приходящих от API в рантайме.
* **Phase 2 – Architecture cleanup (Чистка Архитектуры):**
  - Изолировать логику API от React (создать сервисный слой `services/warehouse.service.ts`).
  - Вынести глобальные состояния в Zustand или React Query. Убить цепочки `useEffect`.
* **Phase 3 – Reliability hardening (Укрепление и Масштабирование):**
  - Создание единого shared-слоя компонентов (Listings, PDF Generators, Subcontract Modals), на которые будут ссылаться все роли (Buyer, Accountant, Director).
